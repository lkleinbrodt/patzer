- **2026-05-02:** **R2 / training upload hardening.** All pushes use **`put_object`** (no `upload_file` / S3Transfer nested threads). Added **`flush_r2_uploads()`** — noop flush via single-worker executor — called when the training loop exits and from **`_final_r2_sync`** atexit so async uploads complete before teardown. **`push_async`**: queue-depth warning when >5 pending, **`submit` failure → synchronous foreground fallback** (never silent drop). Config **`r2_async_uploads`** (default True): set False to block on every large upload. `push_file` / CLI **`push_dir`** now use the same streaming put path.
- **2026-05-02:** `eval/engine.py` — **vectorized legal-move mask** (`_apply_legal_move_mask`) and **`_MoveTokenCache`** for incremental UCI→token ids (append-only play). Regression suite `eval/test_patzer_engine.py` (unittest) compares mask/top-k/cache/`get_move` to reference implementations; run `python -m unittest eval/test_patzer_engine.py -v` from repo root.
- **2026-05-02:** `patzer/r2.py` — **cached** boto3 client (thread-safe) with **adaptive retries** and longer read timeout; **`pull_file` / `pull_dir`** skip only when **`is_fresh`** (ETag sidecar matches R2), otherwise re-download; **push** paths write **`.r2meta`** after upload so pulls can verify freshness without `--force`.
- **2026-05-02:** Training / config / checkpoints — `patzer/configurator.py` parses `--key=value` with `split('=', 1)` and allows `None` defaults to accept any typed override. `patzer/train.py`: `torch.amp.GradScaler('cuda')` (CUDA fp16 only); cosine LR uses a safe denominator + clamped decay ratio; shared `patzer/checkpoint_util.load_checkpoint` (prefer `weights_only=True` + `GPTConfig` safe globals, fallback for legacy). `eval/engine.py` and `patzer/sample.py` use the same loader; `sample.py` context crop matches engine (preserve result token only when present). `CODE_REVIEW.md` deduped.
- **2026-05-02:** `pipeline/filter_games.py` — CLI now **`--allow-bots`** and **`--include-variants`** (opt-in); old `--no-bots` / `--standard-only` never worked as documented (`store_true` + `default=True`). `parse_pgn.py` — progress log moved into **`flush_game()`** so it fires once per N **games**, not once per PGN line. `CODE_REVIEW.md` trimmed to living backlog.
- **2026-05-02:** Replaced stale `patzer/sample.py` (nanoGPT/tiktoken) with a chess-aware sampler: `ChessTokenizer`, legal-move masking, `conditioning` matching eval, optional UCI `start` / `FILE:`, `weights_best.pt` → `ckpt.pt` fallback, vocab_size check. `CODE_REVIEW.md` trimmed to a living backlog; `CLAUDE.md` sample commands updated (`--out_dir=../checkpoints/...` when cwd is `patzer/`).
- **2026-05-02:** **v4 Phase 1 complete; Phase 2 (cooldown) created and launched.** Phase 1 ran WSD with `cooldown_start_iter=None` (constant LR after warmup) until early stopping at iter 145001 (25 evals without ≥0.001 improvement). `ckpt.pt` at iter 140000, val 1.6548, `evals_without_improvement=20`. Created `train_patzer_v4_cooldown.py`: resumes from that checkpoint, `cooldown_start_iter=140000` (decay starts immediately), `cooldown_iters=30000`, `early_stop_patience_evals=0` (disabled — checkpoint had 20/25 patience consumed, would have re-triggered in 5 evals). Phase 2 ran to 170k but val was still dropping linearly, so extended to 220k via `--max_iters=220000`.
- **2026-05-02:** **`auto_cooldown` added to `train.py`.** New WSD flag. When `True`, instead of stopping when early stopping fires, training sets `cooldown_start_iter=iter_num`, extends `max_iters` by `cooldown_iters`, resets patience, and decays LR to `min_lr` — all in one job. `cooldown_start_iter` is now saved in `ckpt.pt` so mid-cooldown resumes restore the correct decay curve. Eliminates the need for manual two-phase configs going forward.
- **2026-05-02:** **W&B resume bug fixed.** `wandb.log({...}, step=iter_num)` was passing explicit step values, which W&B enforces as monotonically increasing. Resuming from an earlier checkpoint than where a run crashed caused all logs to be silently dropped. Fix: removed `step=iter_num`. W&B's internal counter auto-increments (always monotonic); x-axis is driven by `"iter"` via `define_metric`.
- **2026-05-02:** **atexit R2 final-sync error fixed.** Two attempts: (1) `push_file` → `push_async` didn't help — `_executor` was already shut down by the time atexit fired regardless of LIFO ordering. (2) Added `push_file_threadsafe` to `r2.py` using `client.put_object` (single HTTP PUT, no S3Transfer thread pool, no threads at all) — unconditionally safe in atexit context.
- **2026-05-02:** **Metrics R2 sync silently blocked by large weights uploads.** `push_async` uses `max_workers=1`. When a 160MB `weights_best.pt` upload hangs, all queued `metrics.jsonl` uploads pile up behind it — logs show pushes happening but R2 never updates. Fix: `metrics.jsonl` now uses `push_file_threadsafe` (synchronous, <100ms for a ~30KB file). Large files (weights, ckpt) stay async.
- **2026-05-02:** **v4 WSD retrospective.** Stable phase @ 6e-4 plateaued ~145k iters (val 1.6566 at early stop); 30k linear cooldown to 1e-5 dropped val to 1.507 — ~10× faster per iter than the late stable phase. Continued at min_lr for another 23k iters → ~zero further improvement: v4 is on the basin floor. The dramatic cooldown drop is *expected* WSD behavior (DeepSeek/MiniCPM): the stable phase accumulates representation work that cooldown realizes; it is **not** evidence the constant-LR phase was wasted. Bigger signal: 3.3× data → only +0.007 nats over v3 (1.514 → 1.507). v3's "data-bound" gen-gap signal has flipped — **v4 is capacity-bound at 40M params**. v5 should scale architecture, not chase more data or schedule micro-tuning. v5 schedule defaults: keep WSD with constant LR=6e-4, flip `auto_cooldown=True` (avoids the manual two-phase juggling that caused the iter-140k/160k restart blips), bump cooldown ratio from ~20% to ~25–30% of stable length.
