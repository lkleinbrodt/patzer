- **2026-04-30:** Resuming cloud training runs: `launch.py --resume` pulls `checkpoints/` from R2 and runs `train.py --init_from=resume` (resume from `out_dir/ckpt.pt`). To continue a finished run (e.g. v2 stopped at 150k but val loss still falling), bump `**max_iters`** above the prior stop and relaunch with `--resume`. **Do not raise `lr_decay_iters` to match the new `max_iters` unless you intend to:** `get_lr()` only depends on `iter_num` and the *current* `lr_decay_iters` in the config, not on what schedule was used before the checkpoint. If the first job ended near the cosine floor (e.g. `lr_decay_iters=150k` → LR ≈ `min_lr` at 150k) and the second job sets `lr_decay_iters=250k`, then at 150k the cosine ratio moves back into the middle of the curve and **LR jumps ~35×** for v2 hyperparams—train loss recovers but **val loss can spike** right after resume until the schedule cools again. Fix: keep `lr_decay_iters` at the **original** total (stay at `min_lr` for the extension), or use a deliberate warm-restart / new schedule, or add checkpoint-persisted schedule state if we need something fancier.

- **2026-05-02:** **atexit R2 sync error fixed.** `_final_r2_sync` was calling `r2.push_file()` which uses boto3's S3Transfer (internally threaded). During interpreter shutdown, S3Transfer's thread pool is already torn down, causing `RuntimeError: cannot schedule new futures after interpreter shutdown`. Fix: changed to `r2.push_async()` which submits to our own `_executor`. Atexit handlers run LIFO, so `_final_r2_sync` runs before `_executor.shutdown(wait=True)` — our executor is still alive, submits cleanly, then gets drained by the shutdown handler.
- **2026-05-02:** **W&B resume bug fixed.** `wandb.log({...}, step=iter_num)` was passing explicit step values to W&B, which enforces monotonicity on that argument. When resuming from an earlier checkpoint than where a run crashed, all log calls with steps below the previous maximum were silently dropped — the dashboard appeared frozen. Fix: removed `step=iter_num` from `wandb.log()`. W&B's internal counter now auto-increments (always monotonic), and the x-axis on all charts is driven by the `"iter"` field already set up via `define_metric("iter")`. Restarts from any checkpoint will always log correctly going forward.
- **2026-05-02:** **WSD cooldown workflow clarified; `auto_cooldown` added.** Two-phase v4 design was intentional: Phase 1 trained at constant LR until early stop, Phase 2 (`train_patzer_v4_cooldown.py`) resumes with a linear decay. For v5+, added `auto_cooldown = True` flag to `train.py` so this happens automatically in a single job — when early stopping would fire, training instead sets `cooldown_start_iter = iter_num`, extends `max_iters` by `cooldown_iters`, resets patience, and decays to `min_lr` before stopping. `cooldown_start_iter` is now saved in `ckpt.pt` so mid-cooldown restarts restore the correct decay curve without manual config changes. Manual two-phase flow still works (explicit `cooldown_start_iter` in config takes precedence over checkpoint). On cooldown length: 30k iters (~983M tokens, ~20% of the stable phase) is on the generous side — WSD literature typically uses 10%. The bulk of the benefit is front-loaded (LR drops fastest in the first few thousand cooldown iters), so going longer has strongly diminishing returns. The bigger risk of leaving loss on the table is an undertrained stable phase, not an underdone cooldown.
- **2026-05-02:** **v4 Phase 1 complete; Phase 2 cooldown config created.** Phase 1 (`train_patzer_v4.py`) ran WSD with `cooldown_start_iter=None` (constant LR after warmup) until early stopping triggered at iter 145001 (25 evals × 1000 iters without ≥0.001 improvement). `ckpt.pt` landed at iter 140000 with val 1.6548, `evals_without_improvement=20`. Created `train_patzer_v4_cooldown.py` for Phase 2: resumes from that checkpoint, sets `cooldown_start_iter=140000` (decay starts immediately on first resumed step), `cooldown_iters=30000` (linear decay to `min_lr=1e-5` by iter 170000), `early_stop_patience_evals=0` (disabled — Phase 1 left `evals_without_improvement=20` in the checkpoint so early stop would trigger after just 5 more evals otherwise). Also diagnosed the harmless `RuntimeError: cannot schedule new futures after interpreter shutdown` seen in Phase 1 logs: the `atexit` handler tried to enqueue an upload after the thread pool had already been torn down; the weights were already safely on disk at that point.
